# ADR-005: デバイスメモリスクリプトエンジン

## ステータス
**承認済み** (2026-06-20)

## コンテキスト
PLCの実プログラムが動いているかのようなデバイス値の動的変化をシミュレートしたい。  
温度センサーの値変動、カウンタの増加、条件に基づくビットのON/OFF等を簡単に定義できる仕組みが必要。

## 検討した選択肢

### 選択肢1: Python スクリプト直接実行
- **メリット**: 表現力が高い、Pythonの全機能が使える
- **デメリット**: セキュリティリスク（任意コード実行）、学習コスト高

### 選択肢2: 独自DSL
- **メリット**: 完全にコントロール可能、安全
- **デメリット**: パーサー開発コスト高、学習が必要

### 選択肢3: YAML定義 + 安全な式評価
- **メリット**: YAMLは広く知られ可読性が高い、式評価はast.parseでサンドボックス化可能、宣言的で直感的
- **デメリット**: 複雑なロジックには不向き（但しPLCシミュレーション用途では十分）

## 決定
**選択肢3: YAML定義 + 安全な式評価** を採用する。

## スクリプトタイプ

| タイプ | 用途 | 例 |
|:--|:--|:--|
| periodic | 一定間隔で値を更新 | 温度変動、カウンタ |
| conditional | 条件に基づいて値を変更 | アラーム判定 |
| sequence | ステップ順次実行 | 起動シーケンス |
| ramp | 値を段階的に変化 | 圧力上昇 |

## 安全性設計
- `eval()` / `exec()` は使用しない
- `ast.parse()` でPython式をASTに変換し、ホワイトリスト方式で安全なノードのみ許可
- 許可する関数は組み込み関数（sin, cos, random等）に限定
- ファイルシステム・ネットワークアクセス・import文は不可
- 式評価にタイムアウトを設定（無限ループ防止）

## パフォーマンス設計
- スクリプトは独立したasyncioタスクとして実行
- プロトコル応答パスをブロックしない
- デバイスメモリへの書込みはLock経由でスレッドセーフに行う
